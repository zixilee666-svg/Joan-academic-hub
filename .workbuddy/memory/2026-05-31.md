# 2026-05-31 工作日志

## 01. 修复所有 /api/* 路由返回 HTML (commit: dcfae04)

### 问题
commit 82fc74a 将 `edge-functions/api/[[default]].js` 重命名为 `_kv-handler.js` 后，所有 `/api/*` 路由返回 `<!DOCTYPE html>` 而非 JSON，导致首页无法加载 spaces/papers 等数据。

### 根因分析
- `[[default]].js` 在 EdgeOne Pages 中是 catch-all 文件名，自动拦截所有 `/api/*`
- 重命名后，Edge Function 路由配置无法正确匹配，请求 fall-through 到静态页面
- 历次 regex 尝试失败原因：
  - `(?!ai/)` 负向先行断言：EdgeOne 正则引擎不支持
  - 字符类 `a[^i][a-z]*|[b-z][a-z]*`：Edge Function 文件系统 catch-all 仍覆盖 regex

### 修复方案
1. **非 catch-all 文件名**：`edge-functions/api/_api.js`（普通文件名）
2. **pages.config.json 双路由规则**：
   - `^/api/ai/conversations(/.*)?$` → `_api.js`（显式包含：对话需要 KV 存储）
   - `^/api/([^a][^/]*|a[^i][^/]*)(/.*)?$` → `_api.js`（排除 ai/）
3. **Cloud Function 接管**：`cloud-functions/api/ai/[[default]].js` → `/api/ai/chat` + `/api/ai/parse-paper`

### 路由验证结果
| URL | 处理器 |
|------|------|
| `/api/ai/conversations` | Edge Function (_api.js) ✓ |
| `/api/papers/123` | Edge Function ✓ |
| `/api/auth/login` | Edge Function ✓ |
| `/api/ai/chat` | **Cloud Function** ✓ |
| `/api/ai/parse-paper` | **Cloud Function** ✓ |

### 部署
- `dcfae04`: fix: 恢复 API 路由 — _api.js 非 catch-all + 精确 regex 分流
- 推送到 GitHub → EdgeOne Pages 自动部署

## 02. 最终修复：AI 端点迁移至 /api-ai/* (commit: f4ed2a4)

### 问题
- dcfae04 部署后仍返回 HTML（所有 API 路由失效）
- `_api.js` 非标准文件名，EdgeOne 文件系统路由无法识别
- regex 分流彻底失败，所有 API 请求 fall-through 到静态页面

### 根因分析（EdgeOne Pages 路由机制）
1. **Edge Function 使用文件系统路由**：`edge-functions/api/[[default]].js` → `/api/*`
2. **[[default]].js 是 catch-all 文件名**：自动拦截该目录下所有未匹配路由
3. **pages.config.json edgeFunctions.routes 无法覆盖文件系统 catch-all**
4. **Cloud Functions 同样使用文件系统路由**：`cloud-functions/api/ai/[[default]].js` → `/api/ai/*`
5. **冲突时 Edge Function catch-all 优先级高于 Cloud Function**

### 最终方案（路径隔离）
**核心策略：将 AI Cloud Function 部署在不同路径，避免被 Edge Function 拦截**

1. **Edge Function**：`edge-functions/api/[[default]].js` → `/api/*`（CRUD + KV 全部路由）
2. **Cloud Function**：`cloud-functions/api-ai/[[default]].js` → `/api-ai/*`（AI 解析 + 对话）
3. **旧端点安全网**：Edge Function 中 `/api/ai/chat` 和 `/api/ai/parse-paper` 返回 308 重定向到 `/api-ai/*`
4. **前端更新**：`src/lib/api.ts` + `src/services/chatService.ts` 中 AI 端点改为 `/api-ai/chat`、`/api-ai/parse-paper`
5. **pages.config.json**：移除所有 regex 路由配置（文件系统路由自动生效）

### 路由优先级（最终版）
| 路径 | 处理器 | 说明 |
|------|--------|------|
| `/api/*` | Edge Function | 全部 CRUD、KV 存储（V8, 200ms） |
| `/api-ai/*` | Cloud Function | AI 解析 + 流式对话（Node.js, 120s） |
| `/api/ai/chat` | 308 重定向 → `/api-ai/chat` | 旧端点兼容 |
| `/api/ai/parse-paper` | 308 重定向 → `/api-ai/parse-paper` | 旧端点兼容 |

### 部署
- `f4ed2a4`: fix: EdgeOne 路由修复 — AI 端点迁移至 /api-ai/ 路径
- 推送到 origin-new (HTTPS) → EdgeOne Pages 自动部署

## 03. 增强 AI 文献解析质量 (commit: 293e1ed)

### 问题
- AI 解析结果质量差：年份错误（默认2026）、卷号期号为占位符、摘要缺失、信息量少
- 影响后续文献引用和学术分析工作

### Cloud Function 增强 (api-ai/[[default]].js)
1. **增大上下文**：输入截断 3000→8000 字符，maxTokens 1024→4096
2. **增强 System Prompt**：要求完整摘要、学术结构分析（方法/贡献/局限性/结论）
3. **新增字段**：sourceType, citationCount, researchMethod, mainContribution, limitations, conclusion
4. **年份校验**：多模式从原始文本提取年份；不合理时设为 null（不再默认当前年）
5. **增强 regex fallback**：卷号/期号/页码/DOI/URL/关键词/来源类型提取
6. **增强引用格式生成器**：
   - BibTeX：类型推断(article/inproceedings)、字段省略、特殊字符转义
   - IEEE/GB7714：省略空字段，更规范的格式

### 前端增强 (AiParseResultDialog.tsx)
1. **新增字段输入**：sourceType（期刊类型）、citationCount（引用次数）
2. **移除默认年份**：year 改为 number|null，不再默认当前年
3. **移除占位符**："如: 42" → "42"，空值不显示提示
4. **摘要展开/收起**：超过300字符时显示展开按钮
5. **新增学术分析区域**：研究方法、核心贡献、局限性、结论（只读展示）
6. **新图标**：Lightbulb, AlertTriangle, CheckCircle, Microscope, BarChart3
