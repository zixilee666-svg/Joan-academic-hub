## 修复：Dashboard 401 Unauthorized + spaces UNAUTHORIZED（commit: 48b19c0）

### 问题来源
截图显示 Dashboard 页面：
- `/api/stats/reading` → 401 Authorization Required
- `/api/spaces?sort=popularity` → UNAUTHORIZED（`{success:false, error:"Unauthorized"}`）

### 根因分析（5个）

1. **`services/api.ts` `getToken()` 格式错误（核心根因）**
   - 旧代码：`return JSON.parse(localStorage.getItem('joan_auth_token') || 'null')`
   - 问题：Zustand persist 存储格式为 `{state:{token:"eyJ...",user:{...}},version:0}`，直接 `JSON.parse` 返回整个对象，导致 `Authorization: Bearer [object Object]` → 401
   - 修复：提取 `parsed?.state?.token`，兼容旧格式字符串

2. **`spaceService.ts` 非 list 方法无 IS_MOCK 守卫**
   - `getProfile`/`getMaterials`/`getTheme`/`updateConfig`/`recordView` 直接调 `apiClient`（走真实 API），在真实模式下 token 错误 → UNAUTHORIZED
   - 修复：所有方法加 IS_MOCK 判断，Mock 模式走 `libApi` 公共方法

3. **`lib/api.ts` mockRequest spaces list 条件永远为 false**
   - 条件 `!path.includes('/')` 对 `/spaces?...` 永远是 false（path 本身含 `/`）
   - 修复：改为 `!path.match(/\/spaces\/[^?]/)`

4. **`lib/api.ts` mockRequest 缺少 spaces 子路由的 Mock 处理**
   - `/spaces/:username` GET、`/spaces/:username/materials` GET、`/spaces/:username/view` POST、`/spaces/me` PUT 全部没有 Mock handler
   - 修复：补充所有 4 个 handler

5. **`lib/api.ts` ApiClient 没有公共 space 操作方法**
   - `spaceService` 需要通过 `libApi` 调用，但 `request()` 是 private 方法
   - 修复：添加 `getSpaceProfile`/`getSpaceMaterials`/`updateSpaceConfig`/`recordSpaceView` 公共方法

### 修改文件
- `src/services/api.ts` — getToken() 修复
- `src/services/spaceService.ts` — 所有方法加 IS_MOCK 守卫
- `src/lib/api.ts` — mockRequest spaces handlers + ApiClient 公共方法

### 构建状态
- TypeScript: 0 errors
- Vite build: success (~33s)
- Pushed to GitHub: 48b19c0

## 修复：全局 401 鉴权失败处理 + Dashboard 骨架屏卡死（commit: c750d3d）

### 问题来源
截图显示生产环境 Dashboard 页面：
- `/api/stats/reading` → 401（Authorization Required），response: `{success:false, error:"Unauthorized",code:"UNAUTHORIZED"}`
- Dashboard 骨架屏永久加载，无法进入正常页面

### 根因分析
1. **Token 过期或无效（最可能）**：JWT 7 天有效期，或之前 Mock 模式留下的 `mock-jwt-token-...` 在真实模式下被后端拒绝
2. **DashboardPage `loadData` 阻塞**：`getReadingStats()` 抛出 → catch 块捕获 → `setStatsLoaded(true)` 从未执行 → 骨架屏永驻
3. **缺乏全局 401 处理**：任何 API 收到 401 都只是抛出错误，没有自动登出/重定向

### 修复方案
1. **AuthContext.tsx — Token 启动验证**：
   - 新增 `isTokenExpired()` / `isValidToken()` 辅助函数
   - `getInitialState()` 启动时验证 token 格式（必须 `eyJ` 开头）和过期时间
   - 无效 token 自动清理 localStorage，强制重新登录

2. **AuthContext.tsx — 全局 401 拦截器**：
   - `useEffect` 注册 `addErrorInterceptor`
   - 拦截到 `ApiErrorCode.UNAUTHORIZED` 时：
     - 清理 localStorage / Zustand
     - 设置 `sessionStorage.setItem('joan_just_logged_out', 'true')`
     - 延迟 100ms 跳转 `window.location.hash = '#/login'`
   - 使用 `useRef` 防止重复触发

3. **DashboardPage.tsx — 统计独立容错**：
   - `getReadingStats()` 单独包一层 try/catch
   - `setStatsLoaded(true)` 放在外层 try，确保无论统计是否成功都会解除骨架屏
   - `ensurePapers()` / `ensureProjects()` 不受影响

### 修改文件
- `src/context/AuthContext.tsx` — Token 验证 + 401 拦截器
- `src/pages/DashboardPage.tsx` — 统计独立容错

### 构建状态
- TypeScript: 0 errors
- Vite build: success (~2.4s)
- Pushed to GitHub: c750d3d

## 修复：401 拦截器竞态 — 模块级同步注册（commit: 820e9ba）

### 问题来源
截图显示生产环境 APIs 全部 401（papers/reading/spaces），但没有触发登录页跳转。
Console 中没有 `[Auth] 检测到 401` 日志。

### 根因分析
**竞态条件**：`AuthContext` 在 `useEffect` 中注册 401 错误拦截器，但 `DashboardPage` 的 `loadData()` 也在 `useEffect` 中发起 API 请求。虽然 React 18 规定父组件 effect 先执行，但存在不确定性：
- 如果 `loadData()` 在拦截器注册之前发起——拦截器未注册 → 401 错误直接传到 Dashboard catch → 无跳转
- 三个 API 调用同时 401 → 证明拦截器确实未生效

### 修复方案
1. **`lib/api.ts` — 模块级同步拦截器**（核心修复）：
   - 在 `addErrorInterceptor()` 定义之后立即注册，位于模块顶层（非 React 生命周期内）
   - `import` 时即注册，早于任何组件挂载
   - 用 `_401redirected` 模块级标志防重复跳转
   - 同步清理 localStorage，立即 `window.location.hash = '#/login'`（不用 setTimeout）

2. **`AuthContext.tsx` — 简化为启动检查**：
   - 移除 `useEffect` 中的重复拦截器
   - 新增启动后立即检查：`!isAuthenticated && hash !== '#/login'` → 直接跳转

### 修改文件
- `src/lib/api.ts` — +18 行，模块级 401 拦截器
- `src/context/AuthContext.tsx` — 移除 useEffect 拦截器，新增启动检查

- Vite build: success (~2.2s)
- Pushed to GitHub: 820e9ba

## 修复：POST /api/import 405 Method Not Allowed（commit: b670da4）

### 问题来源
截图显示 ImportExportPage 点击"导入"后：
- `POST /api/import` → 405 Method Not Allowed
- Response: `{"success":false,"message":"Method not allowed","version":"v3-retry-20260529"}`
- Console: `[Import] Import error: ApiError: Method not Allowed`

### 根因分析
Edge Function 中 `handleImportFromSearch` 函数已定义但**从未被路由调度**——它是孤立的。
- `POST /api/import/zotero` 有路由 ✅
- `POST /api/search/import` 无路由 ❌（前端 `api.importFromSearch()` 调用此路径）
- `POST /api/import` 无路由 ❌（legacy/备用路径）

两个路径都 fall through 到默认 "Method not allowed" 响应。

### 修复方案
在 Zotero import 路由后添加两个路由：
1. `POST /api/search/import` → `handleImportFromSearch`（前端实际路径）
2. `POST /api/import` → `handleImportFromSearch`（兼容路径）
