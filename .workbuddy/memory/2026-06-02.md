## 修复 arXiv + Semantic Scholar 搜索 404 (commit: e102701)

**问题**: 导入导出页面搜索无结果，Network 显示 `No data found for resource with given identifier`，Console 报 `search_error: ApiError`

**根因**: 前端调用 `/api-external/search/arxiv` 和 `/api-external/search/semantic-scholar`，但 EdgeOne Pages 上该路径没有 edge function handler：
- `edge-functions/api/[[default]].js` 只处理 `/api/*` 路由
- `/api-external/*` 匹配 `pages.config.json` 的 `/* → ./dist` catch-all → 返回 `index.html`
- HTML 作为 JSON 解析失败 → 搜索报错

**修复 (3 层)**:
1. **前端路径修正**: `api.ts` 中 `searchArxiv` / `searchSemanticScholar` 路径从 `/api-external/search/*` → `/search/*`（`realRequest` 自动添加 `baseUrl=/api` → `/api/search/*`）
2. **Edge Function 路由**: 在 `segments[0] === 'search'` 处分发到 `handleSearchArxiv` / `handleSearchSemanticScholar`
3. **代理处理函数** (新增 ~180 行):
   - `handleSearchArxiv`: 代理 `export.arxiv.org/api/query`，解析 Atom XML → JSON
   - `handleSearchSemanticScholar`: 代理 `api.semanticscholar.org/graph/v1/paper/search`，透传 JSON
   - 辅助函数: `extractXmlTag`, `extractXmlAuthors`, `extractXmlYear`, `extractXmlLink`

**技术细节**:
- arXiv API 15s 超时，返回 Atom XML → 服务端解析为 JSON
- Semantic Scholar API 15s 超时，支持可选 `x-api-key` header
- ambos 使用 `User-Agent: JoanAcademicHub/1.0`
- TypeScript 零错误，Vite build 1.69s

## 修复 AI 模型名称误填为 URL + arXiv 搜索超时 (commit: 1ada8da)

**问题1 (截图2/3)**: AI 智能解析与 AI 对话均返回 400 错误：
`The supported API model names are 'deepseek-v3-pro' or 'deepseek-v4-flash', but you passed https://api.deepseek.com`

**根因**: 用户将 API URL (`https://api.deepseek.com`) 误填为「模型名称」字段，DeepSeek API 收到非法 model 参数后拒绝请求。

**修复 (3 层防护)**:
1. **前端 aiParser.ts**: `parsePaperDirectly` 在调用 AI API 前检查 `model`，若为空或以 `http` 开头则自动回退到 `deepseek-chat`
2. **Cloud Function `api-ai/[[default]].js`**: `handleAiChat` 和 `handleParsePaper` 均添加相同校验
3. **Edge Function `api/[[default]].js`**: `handleAiChat` 同样添加校验

**问题2 (截图1)**: arXiv 搜索请求超时（Network 红色，Console: "请求超时，请重试"）

**根因**: Edge Function 第二套 `handleSearchArxiv`（2757行，含缓存逻辑，实际生效版本）未设置 fetch 超时，arXiv API 响应慢时前端 30s 超时触发。

**修复**:
- arXiv 搜索 fetch 添加 `AbortSignal.timeout(20000)`（原无超时）
- Semantic Scholar 搜索 fetch 同样添加 20s 超时（原无超时）

**构建**: Vite build 1.64s，零错误，已推送至 main

## 修复 arXiv + Semantic Scholar 搜索 545 错误 — 前端直连绕过 Edge Function (commit: 8681c36)

**问题 (截图1/2)**: arXiv 和 Semantic Scholar 搜索均返回 545 (Unknown Status)，左侧显示 Mock 数据

**根因**: EdgeOne Pages 的 Edge Function 运行在 V8 isolate 中，**无法访问外部网络**。`fetch()` 调用 `export.arxiv.org` 或 `api.semanticscholar.org` 时失败，EdgeOne 平台返回 545 错误。

代码层面：Edge Function `edge-functions/api/[[default]].js` 第 2344-2346 行拦截了 `/api/search/*` 请求，但 handler 内部的 fetch 外网调用必然失败。

**修复（与 parsePaper 超时相同策略 — 前端直连）**:

1. **src/lib/api.ts — 前端直连 arXiv API**:
   - `searchArxiv()` 真实模式下直接 `fetch('https://export.arxiv.org/api/query?...')`
   - 浏览器端解析 Atom XML（`extractXmlTag`/`Authors`/`Year`/`Link` 辅助函数）
   - 20 秒超时控制
   - Mock 模式保留原有逻辑

2. **src/lib/api.ts — 前端直连 Semantic Scholar API**:
   - `searchSemanticScholar()` 真实模式下直接 `fetch('https://api.semanticscholar.org/graph/v1/paper/search?...')`
   - 解析 JSON 响应，统一字段映射
   - 支持 `x-api-key` header 透传
   - Mock 模式保留原有逻辑

3. **Edge Function 清理**:
   - 删除第 2344-2346 行 `/search/*` 路由拦截（不再被调用）
   - 删除第 3046-3058 行注释式透传逻辑（无实际代码）

**技术说明**:
- arXiv API 支持 CORS (`Access-Control-Allow-Origin: *`)，浏览器可直接访问
- Semantic Scholar API 同样支持 CORS
- 前端直连避免了 EdgeOne Pages Edge Function 的网络沙箱限制
- 同时保留了 Cloud Functions 的搜索代码 (`cloud-functions/api/search/[[default]].js`) 作为备份

**构建**: TypeScript 零错误，Vite build 1.67s，已推送至 main

## 优化 arXiv 搜索稳定性 — 多域名备用+重试+AbortController (commit: c7b5e1e)

**问题 (截图)**: arXiv 搜索仍返回 `TimeoutError: signal timed out`，20 秒超时不够；快速切换搜索源时存在请求竞争

**根因分析**:
1. `export.arxiv.org` 在国内访问慢，单域名无备用
2. 无重试机制，一次失败即放弃
3. 每页 10 条导致响应体大、传输慢
4. 用户快速切换搜索源/输入时，旧请求未取消，造成竞争

**修复（借鉴 arxiv-reader skill 最佳实践）**:

1. **src/lib/api.ts — 新增 `fetchArxivWithRetry` 辅助函数**:
   - 双域名备用：`export.arxiv.org` (主) → `arxiv.org` (备)
   - 主域名重试 2 次，备域名 1 次
   - 指数退避：1s → 2s → 4s（最大）
   - 超时从 20s 增至 30s（与 skill 的 `requests.get(timeout=30)` 一致）
   - 支持外部传入 `AbortSignal`

2. **src/lib/api.ts — `searchArxiv` 重构**:
   - 默认 `maxResults` 从 10 降至 5（首次加载更快）
   - 新增可选 `signal?: AbortSignal` 参数
   - 内部调用 `fetchArxivWithRetry` 替代直接 `fetch`

3. **src/pages/ImportExportPage.tsx — AbortController**:
   - 新增 `abortControllerRef`，组件卸载时自动 abort
   - `handleSearch` 开始时取消旧请求，传入 signal
   - `handleLoadMore` 同样传入 signal
   - catch 块中静默处理 `AbortError`（不弹 toast）

**构建**: TypeScript 零错误，Vite build 21.76s，已推送至 main

## 修复 arXiv 搜索 — CORS 代理回退 + Mock 优雅降级 (commit: 63735e6)

**问题 (截图)**: arXiv 搜索仍返回 `Failed to fetch` / `All retries failed`，左侧显示 Mock 假数据。Semantic Scholar 正常。

**根因深度分析**:
1. 浏览器直连 `export.arxiv.org` 在用户网络下完全不可达（DNS/防火墙/GFW）
2. 原备选域名 `arxiv.org` 对 `/api/query` 返回 302 重定向到 `export.arxiv.org`，毫无意义
3. 两个域名实际指向同一目标服务器，同时失效

**修复**:

1. **src/lib/api.ts — `fetchArxivWithRetry` 重构**:
   - 移除无意义的 `https://arxiv.org` 备选（只会 302 重定向）
   - 新增 CORS 公共代理回退：`https://corsproxy.io/?https://export.arxiv.org`
   - 代理超时 45s（额外延迟开销）
   - 代理返回 JSON 包装时自动提取 `contents` 字段

2. **src/lib/api.ts — `searchArxiv` 优雅降级**:
   - catch 块不再抛出错误，而是返回 Mock 数据 + `_notice` 提示
   - 用户至少能看到结果列表，而非空白页面或 toast 报错
   - 提示信息：`arXiv 服务当前不可用（网络限制），已返回演示数据。建议切换至 Semantic Scholar 搜索。`

3. **src/lib/api.ts — `generateMockArxivResults`**:
   - 新增包装器，将 `generateMockSearchResults('arxiv')` 的字段映射为真实 arXiv API 格式（`pdfUrl`/`abstractUrl`/`published`/`updated`/`summary`）

**构建**: TypeScript 零错误，Vite build 1.60s，已推送至 main

## 修复设置跨设备同步 — 自动拉取+统一保存+KV 鲁棒性 (commit: a90e372)

**问题**: 一台设备保存 API Keys / AI 模型后，另一台设备登录仍需要重新填写。

**根因（多层）**:
1. `saveProfile`（个人资料"保存修改"按钮）只保存 theme/citationFormat/notifications，**不包含外部工具字段**。若用户先点"保存修改"再用 KV getSettings 拉取，外部工具字段（zoteroUserId 等）可能被 DEFAULT_SETTINGS 的空字符串覆盖
2. 外部工具 Tab 需**单独点击"同步到云端"**按钮，用户容易遗漏
3. SettingsPage **不会主动从 KV 拉取**最新配置，依赖 AuthContext 登录时的一次性拉取
4. Edge Function `handleGetSettings` 中 `settings || DEFAULT_SETTINGS` 若 settings 为 `{}`（空对象，truthy）会返回空数据

**修复**:

1. **SettingsPage** — 打开时自动从云端拉取：
   - 新增 `useEffect`，mount 时调用 `api.getSettings()` → 检测云端是否有实质数据 → `loadFromBackend`
   - 仅当云端有数据时才合并（避免 DEFAULT_SETTINGS 覆盖本地配置）

2. **SettingsPage** — 统一保存（一站式）：
   - `saveProfile` 现在同时发送外部工具字段（zoteroUserId/apiKey、githubToken、imaApiKey、aiModels 等）
   - 用户**只需点一次"保存修改"**，所有配置都会同步到 KV

3. **SettingsPage** — 新增「从云端拉取」按钮：
   - 外部工具 Tab 底部新增"从云端拉取"按钮（与"保存配置"并列）
   - 手动触发 KV 拉取，带 toast 反馈

4. **Edge Function handleGetSettings** — 防御性校验：
   - `settings || DEFAULT_SETTINGS` → 先检测 settings 是否为非空对象
   - `{ ...DEFAULT_SETTINGS, ...settings }` 保证所有字段都有默认值
   - 新增日志：`fromKV` 和 `keys` 数量

5. **Edge Function handleUpdateSettings** — 增强日志：
   - 记录 `bodyKeys` 和 `totalKeys`，方便排查保存内容

6. **userStore.ts loadFromBackend** — 改进合并逻辑：
   - `aiModels` 的 `apiKey` 字段确保为字符串（`m.apiKey || ''`）
   - 改进日志输出

**构建**: TypeScript 零错误，Vite build 1.72s，已推送至 main
